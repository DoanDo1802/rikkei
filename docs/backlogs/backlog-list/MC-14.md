# Mã Backlog: MC-14 – Biểu đồ thống kê và Báo cáo (Admin)
## 1. Mô tả chi tiết & Yêu cầu kỹ thuật
**Mục tiêu:** Xây dựng hệ thống báo cáo tổng hợp và trực quan hóa dữ liệu (Data Visualization) dành cho quản trị viên. Hệ thống sẽ chuyển đổi hàng nghìn bản ghi thô từ cơ sở dữ liệu PostgreSQL thành các chỉ số vận hành then chốt (KPIs) và biểu đồ xu hướng, giúp Admin nắm bắt nhanh tình hình hoạt động của phòng khám và đưa ra các quyết định điều phối nhân sự hoặc nhập kho thuốc chính xác.
### Thư viện / Công nghệ khuyến nghị
- **Backend**: Python (FastAPI) sử dụng các hàm gom nhóm (GROUP BY, COUNT, DISTINCT) của SQLAlchemy hoặc SQLModel để tối ưu tốc độ truy vấn.
- **Frontend**: Next.js tích hợp thư viện Chart.js hoặc Recharts để vẽ biểu đồ.
- **Xử lý dữ liệu**: Sử dụng date-fns hoặc moment để xử lý các khoảng thời gian báo cáo (theo tuần, tháng, năm).
### Yêu cầu kỹ thuật cốt lõi
- **Aggregation Logic (Logic gom nhóm)**:
- **Thống kê lượt khám (Visits)**: Đếm tổng số bản ghi trong bảng medical_records (Mỗi bản ghi EMR tương ứng 1 lượt khám).
- **Thống kê Bệnh nhân (Unique Patients)**: Đếm số lượng patient_id duy nhất (DISTINCT) đã từng phát sinh ca khám để phân biệt giữa "số lượt đi khám" và "số người đi khám".
- **Phân bổ chuyên khoa**: Thống kê tỷ lệ ca khám theo từng specialty_id để biết khoa nào đang là thế mạnh hoặc bị quá tải.
- **Trực quan hóa (Data Visualization)**:
- **Biểu đồ Cột (Bar Chart)**: Hiển thị số lượng ca khám theo từng ngày trong tuần/tháng để theo dõi biến động lưu lượng.
- **Biểu đồ Tròn (Pie Chart)**: Hiển thị tỷ trọng bệnh nhân giữa các chuyên khoa.
- **Thẻ chỉ số (KPI Cards)**: Hiển thị các con số tổng quát ở đầu trang (Tổng bệnh nhân, Tổng doanh thu dự kiến, Tỷ lệ hủy lịch).
- **Bộ lọc (Filtering)**: Cho phép Admin lọc dữ liệu theo khoảng thời gian (Từ ngày... Đến ngày...).
## 2. Tiêu chí hoàn thành (Definition of Done - DoD)
- Thiết lập thành công API /api/v1/admin/statistics trả về dữ liệu tổng hợp dạng JSON.
- **Tính chính xác**: Số liệu trên Dashboard phải khớp 100% với kết quả truy vấn trực tiếp trong Database (SQL Query).
- **Hiển thị biểu đồ**: Biểu đồ tròn và biểu đồ cột hiển thị đúng tỷ lệ, có chú thích (Legend) rõ ràng và hiệu ứng khi di chuột (Tooltip).
- **Hiệu năng**: Các truy vấn thống kê phải được tối ưu (sử dụng Index trên các cột created_at và specialty_id), thời gian phản hồi API phải dưới 2 giây kể cả khi dữ liệu lớn.
- **Tính năng lọc**: Dashboard cập nhật lại dữ liệu và biểu đồ ngay khi Admin thay đổi khoảng thời gian lọc.
- **Giao diện**: Dashboard được thiết kế chuẩn Admin-UI, hiển thị tốt trên cả màn hình Desktop và Tablet.
## 3. Đặc tả dữ liệu (Data Schema cho API)
- **Cấu trúc dữ liệu JSON trả về cho Frontend sẽ bao gồm**:
| Tên trường | Kiểu dữ liệu | Ý nghĩa |
| :--- | :--- | :--- |
| **Logic SQL (Gợi ý)** | `total_visits` | `Integer` |
| **Tổng số lượt khám (Số ca).** | COUNT(emr_code) | `total_patients` |
| **Integer** | Tổng số bệnh nhân thực tế (Số người). | COUNT(DISTINCT patient_id) |
| **visits_by_day** | Array[Object] | Dữ liệu biểu đồ cột (Ngày & Số ca). |
| **GROUP BY DATE(created_at)** | `specialty_distribution` | Array[Object] |
| **Dữ liệu biểu đồ tròn (Tên khoa & %).** | GROUP BY specialty_id | `top_diseases` |
| **Array[Object]** | Top 5 loại bệnh phổ biến nhất. | GROUP BY diagnosis_icd10 |


## 4. Luồng xử lý kỹ thuật (Technical Flow)
1. Admin truy cập vào trang Dashboard.
2. Frontend (Next.js) gửi yêu cầu kèm theo tham số ngày tháng (Vd: ?start_date=2024-01-01&end_date=2024-01-31).
3. Backend (FastAPI) thực hiện các câu lệnh SQL Aggregation trên Postgres để lấy số liệu tổng hợp.
4. Backend đóng gói dữ liệu vào một Object JSON duy nhất và trả về.
5. Frontend nhận dữ liệu, truyền vào các Component biểu đồ (Chart.js) để render ra giao diện.
### Lưu ý
Đối với dữ liệu cực lớn (hàng triệu bản ghi), cần cân nhắc sử dụng Materialized Views trong Postgres để lưu trữ dữ liệu thống kê đã tính toán sẵn nhằm tăng tốc độ hiển thị Dashboard.
Đảm bảo chỉ có tài khoản có role = ADMIN mới được phép truy cập vào API này.
