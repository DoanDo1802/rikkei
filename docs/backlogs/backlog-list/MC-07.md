# Mã Backlog: MC-07 – Thiết lập Gateway WebSocket truyền tin Thời gian thực
## 1. Mô tả chi tiết & Yêu cầu kỹ thuật
**Mục tiêu:** Xây dựng hạ tầng truyền tin hai chiều (Full-duplex) dựa trên giao thức WebSocket để cập nhật trạng thái hàng chờ và gửi thông báo tức thời. Đảm bảo bác sĩ và bệnh nhân luôn nhận được thông tin mới nhất (như lệnh gọi khám, thay đổi số thứ tự) mà không cần tải lại trang, tạo ra luồng vận hành mượt mà cho phòng khám.
### Thư viện / Công nghệ khuyến nghị
- **Backend**: FastAPI (Python) kết hợp với WebSockets hoặc Broadcaster. Nếu theo đúng luồng STOMP như mô tả: Sử dụng thư viện hỗ trợ STOMP over WebSockets.
- **Message Broker**: Redis Pub/Sub (khuyến nghị để xử lý truyền tin giữa các worker nếu hệ thống mở rộng).
- **Frontend**: SockJS-client và Stompjs (JavaScript) để duy trì kết nối ổn định trên trình duyệt.
### Yêu cầu kỹ thuật cốt lõi
- **Cấu hình Gateway**:
Thiết lập Endpoint /ws yêu cầu xác thực qua JWT ngay tại bước Handshake để đảm bảo an toàn.
- **Phân chia các kênh truyền tin (Topics/Channels)**:
/topic/queue: Kênh chung cho toàn bộ bác sĩ/lễ tân để theo dõi thay đổi hàng chờ real-time.
/user/{patient_id}/notification: Kênh riêng tư cho từng bệnh nhân để nhận thông báo "Đã đến lượt khám" hoặc "Lịch hẹn bị hủy".
- **Cơ chế truyền tin (Pub/Sub)**:
Khi API thay đổi trạng thái appointments.status (từ MC-08), hệ thống phải kích hoạt một sự kiện (Event) gửi Message tới các Topic tương ứng.
- **Tính ổn định (Resilience)**:
Triển khai cơ chế Heartbeat để kiểm tra trạng thái kết nối giữa Client và Server.
Xử lý lỗi ngắt kết nối đột ngột (Network flapping) bằng thuật toán Exponential Backoff cho phía Frontend khi thực hiện kết nối lại.
## 2. Tiêu chí hoàn thành (Definition of Done - DoD)
Thiết lập thành công WebSocket Server tích hợp trong Backend FastAPI/Spring Boot.
- **Xác thực**: Chặn thành công các kết nối WebSocket không kèm theo Token hợp lệ.
Độ trễ (Latency): Thông báo từ lúc bác sĩ nhấn "Gọi khám" đến khi màn hình bệnh nhân hiển thị phải dưới 1 giây (trong điều kiện mạng ổn định).
- **Tính năng Auto-reconnect**: Thực hiện ngắt mạng thủ công và xác nhận Frontend tự động kết nối lại thành công sau khi có mạng trở lại.
- **Tính chính xác**: Đảm bảo Message gửi vào kênh /user/{patient_id}/... chỉ bệnh nhân đó nhận được, không bị rò rỉ sang bệnh nhân khác.
- **Tài liệu hóa**: Viết hướng dẫn tích hợp WebSocket Client cho các thành viên làm Frontend (Subscribe vào kênh nào, xử lý Event ra sao).
## 3. Đặc tả dữ liệu (Message Schema)
- **Mỗi tin nhắn truyền qua WebSocket sẽ có cấu trúc JSON như sau**:
| Tên trường | Kiểu dữ liệu | Ý nghĩa |
| :--- | :--- | :--- |
| **event_type** | `String` | Loại sự kiện (Vd: QUEUE_UPDATED, PATIENT_CALLED). |
| **appointment_id** | Long/UUID | ID của lịch hẹn có sự thay đổi. |
| **patient_id** | `String` | Mã bệnh nhân liên quan (patient_code). |
| **new_status** | `Enum` | Trạng thái mới: WAITING, IN_PROGRESS, DONE. |
| **message** | `String` | Nội dung hiển thị (Vd: "Mời bệnh nhân Nguyễn Văn A vào phòng 101"). |
| **timestamp** | `Long` | Thời gian gửi tin nhắn (Unix timestamp). |


## 4. Phân luồng kỹ thuật (Technical Logic)
1. Frontend (Patient/Doctor): Ngay khi đăng nhập thành công, gọi hàm connect() tới Gateway.
2. Backend: Lưu trữ thông tin kết nối (Session) của người dùng vào bộ nhớ đệm (thường là Redis).
3. Trigger: Bác sĩ gọi số -> API Update DB -> API gọi hàm broadcast_to_user(patient_id, message).
4. Gateway: Tìm kiếm Session của patient_id và đẩy Message xuống thông qua kết nối đang mở.
5. Frontend (Patient): Nhận Event PATIENT_CALLED -> Hiển thị thông báo (Toast/Popup) trên trình duyệt/mobile.

### Lưu ý
Tránh gửi quá nhiều dữ liệu qua WebSocket (chỉ gửi ID và Status). Frontend nên dùng ID đó để fetch thêm dữ liệu nếu cần, nhằm tối ưu băng thông.
Cấu hình CORS cho WebSocket để cho phép Frontend kết nối từ domain khác.
